## 二、进度计划

### 2.1 项目里程碑

```
第1周 ───┬── 第2周 ───┬── 第3周 ───┬── 第4周 ───┬── 第5周
         │            │            │            │
    ★M1  │       ★M2  │       ★M3  │       ★M4  │  ★M5
  需求确认│     后端完成 │    前后端   │    测试完成  │  项目交付
  设计完成│     前端框架 │    联调完成 │    文档完成  │  答辩准备
```

| 里程碑 | 时间 | 交付物 | 验收标准 |
|--------|------|--------|---------|
| **M1** 需求与设计 | 第 1 周末 | 需求说明书、系统设计文档、数据库 DDL | 团队评审通过 |
| **M2** 核心开发 | 第 2 周末 | 后端全部 API、前端框架+公共组件 | 所有端点 curl 可调用 |
| **M3** 前后端联调 | 第 3 周末 | 完整可运行系统、模拟数据填充 | 四角色全流程走通 |
| **M4** 测试与文档 | 第 4 周末 | 测试报告、全部文档初稿 | 缺陷全部关闭 |
| **M5** 交付答辩 | 第 5 周中 | 最终版文档、PPT、源码 | 答辩通过 |

## 三、协作规范

### 3.1 代码管理

- **分支策略**：`1` (主分支) + `feature/*` (功能分支)
- **提交规范**：`<type>: <简短描述>` (feat/fix/docs/refactor/test)
- **合并流程**：功能分支 → PR → 组长评审 → 合并

### 3.2 沟通机制

| 机制 | 频率 | 时长 |
|------|------|------|
| 每日站会 | 每天 9:00 | 10 分钟 |
| 进度同步 | 每天 18:00 | 5 分钟 |
| 周报 | 每周五 | 书面提交 |

### 3.3 文档规范

- 所有文档使用 Markdown 格式
- 图片使用 Mermaid 或 ASCII Art
- 版本号格式：`v<major>.<minor>.<patch>`

## 四、风险管理

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| MySQL C API 编译问题 | 高 | 高 | 提前验证 MinGW + libmysql 兼容性 |
| C 语言缺少现代特性 (无 GC/异常) | 中 | 高 | 统一内存管理规范、每处 malloc 标注 free 责任人 |
| 前后端字段名不一致 | 高 | 中 | API 文档先行、字段名统一用 snake_case |
| 时间不足 | 中 | 高 | 组长可调配 15% 弹性时间 |
| 数据库字符集问题 | 中 | 中 | 统一 utf8mb4，所有 INSERT 前 SET NAMES |
