### 3.3 关键表详细设计

#### users 表

```sql
CREATE TABLE users (
    user_id     INT PRIMARY KEY AUTO_INCREMENT,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    password    CHAR(64)     NOT NULL COMMENT 'SHA256哈希',
    real_name   VARCHAR(50)  NOT NULL,
    role        ENUM('school_admin','college_admin','club_admin','club_member','general_student') NOT NULL DEFAULT 'general_student',
    college_id  INT,
    major_id    INT,
    class_id    INT,
    student_no  VARCHAR(20),
    id_card     VARCHAR(18),
    phone       VARCHAR(20),
    email       VARCHAR(100),
    status      TINYINT      NOT NULL DEFAULT 1 COMMENT '0禁用 1正常 2限制',
    last_login  DATETIME,
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (college_id) REFERENCES colleges(college_id),
    FOREIGN KEY (major_id)   REFERENCES majors(major_id),
    FOREIGN KEY (class_id)   REFERENCES classes(class_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### reimbursement 表 (二级审批)

```sql
CREATE TABLE reimbursement (
    reimbursement_id  INT PRIMARY KEY AUTO_INCREMENT,
    club_id           INT NOT NULL,
    applicant_id      INT NOT NULL,
    amount            DECIMAL(12,2) NOT NULL,
    description       VARCHAR(500),
    receipt_path      VARCHAR(255),
    status            ENUM('pending','approved','rejected') NOT NULL DEFAULT 'pending',
    college_reviewed  ENUM('pending','approved','rejected') NOT NULL DEFAULT 'pending',
    review_comment    VARCHAR(500),
    submitted_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    reviewed_at       DATETIME,
    FOREIGN KEY (club_id)      REFERENCES clubs(club_id),
    FOREIGN KEY (applicant_id) REFERENCES users(user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.4 触发器 (5 个)

| 触发器 | 触发时机 | 功能 |
|--------|---------|------|
| `trg_member_after_insert` | members INSERT | 自动 `clubs.member_count + 1` |
| `trg_member_after_update` | members UPDATE | 成员状态变化时同步 member_count |
| `trg_registration_after_insert` | registrations INSERT | 自动 `activities.current_count + 1` |
| `trg_registration_after_update` | registrations UPDATE | 取消报名时 `current_count - 1` |
| `trg_reimbursement_after_update` | reimbursement UPDATE | 报销批准时累加已使用额度 |

## 四、API 接口设计

### 4.1 通用规范

```json
// 成功响应
{"code": 0, "msg": "ok", "data": {...}}

// 列表响应 (分页)
{"code": 0, "msg": "ok", "data": {"list": [...], "total": 50, "page": 1, "page_size": 10}}

// 错误响应
{"code": -8, "msg": "用户名或密码错误"}
```

### 4.2 返回码定义

| 码 | 常量 | 含义 |
|----|------|------|
| 0 | OK | 成功 |
| -1 | ERR_DB | 数据库错误 |
| -2 | ERR_NOT_FOUND | 记录不存在 |
| -3 | ERR_DUPLICATE | 重复记录 |
| -4 | ERR_PERMISSION | 权限不足 |
| -5 | ERR_VALIDATION | 数据校验失败 |
| -6 | ERR_FULL | 名额已满 |
| -7 | ERR_STATUS | 状态不允许 |
| -8 | ERR_AUTH | 认证失败 |
| -9 | ERR_INPUT | 输入参数错误 |
| -10 | ERR_QUOTA_EXCEED | 超出额度 |
| -11 | ERR_RESTRICTED | 用户被限制 |
| -12 | ERR_ELECTION | 选举流程错误 |
| -13 | ERR_VOTE_TIED | 平票 |
| -99 | ERR_SYSTEM | 系统级错误 |

### 4.3 路由匹配机制

URI 模式支持 `*` 通配**单个路径段**：

| 模式 | 匹配 | 不匹配 |
|------|------|--------|
| `/api/school/clubs/*` | `/api/school/clubs/12` | `/api/school/clubs/12/approve` |
| `/api/club/*/activities/*` | `/api/club/1/activities/5` | `/api/club/1/activities` |

Handler 通过 `api_get_path_int(ctx, idx)` 按 `/api/` 后的段索引读取参数。

| 模块 | 数量 | 说明 |
|------|------|------|
| 公共接口 (无需登录) | 4 | 注册、学院/专业/班级列表 |
| 通知中心 | 4 | 列表、未读数、已读标记 |
| 学校端 | 27 | 仪表盘/社团审批/社团管理/用户管理/财务监督/统计/公告/配置/备份 |
| 学院端 | 17 | 仪表盘/班级/社团监管/财务审批/学生管理/公告 |
| 学生端 | 20 | 社团广场/活动/个人中心/我的社团 |

### 4.5 认证流程

```
POST /api/login {username, password}
  → auth_login() 校验密码 (SHA256)
  → 更新 last_login
  → 生成 32 位随机 Token
  → Set-Cookie: scms_token=<token>; HttpOnly; SameSite=Lax
  → 返回 user_id, real_name, role

后续请求:
  → Cookie 中取 scms_token
  → auth_resolve_session() 查内存哈希表
  → auth_load_user() 从 DB 重新加载 (保证实时性)
  → 填充 ctx->user
```

## 六、核心业务状态机

### 6.1 活动状态机

```
  draft ──publish──→ published ──start──→ ongoing ──finish──→ completed
    │                     │                   │
    └────── cancel ───────┴───────────────────┘
                              │
                              ▼
                          cancelled (终态)
```

### 6.2 报销审批链

```
社长提交 (status=pending, college_reviewed=pending)
  │
  ▼
学院管理员审批
  ├── 驳回 → status=rejected, college_reviewed=rejected
  └── 通过 → college_reviewed=approved
      │
      ▼ (仅院级社团)
  学校管理员终审
      ├── 驳回 → status=rejected
      └── 通过 → status=approved → INSERT INTO finance(type='expense')
```

### 6.3 选举流程

```
创建选举 (signup阶段)
  ├── 社长连任申请
  ├── 成员报名参选
  │
  ▼
投票阶段 (voting)
  ├── 成员匿名投票 (每人每职位 1 票)
  │
  ▼
选举结束 (completed)
  ├── 统计票数 → 更新 members.role + clubs.president_id
  └── 平票 → 需重投或管理员裁决
```

## 七、安全设计

| 层面 | 措施 |
|------|------|
| **密码存储** | SHA256 哈希，不可逆，64 位十六进制 |
| **会话** | 32 位随机 Token，HttpOnly Cookie，SameSite=Lax |
| **SQL 注入** | 所有字符串参数通过 `db_escape()` (mysql_real_escape_string) 转义，数值用 `%d` 格式化 |
| **文件上传** | 随机文件名 (时间戳+6 位随机码)，扩展名白名单，目录穿越防护 |
| **权限** | 路由级 `need_login` + 业务级 `api_require_*` + 数据行级归属校验 |
| **CORS** | OPTIONS 预检返回 `Access-Control-Allow-Origin: *` |

## 九、关键设计决策

| # | 决策 | 原因 |
|---|------|------|
| 1 | 单体二进制部署 | 免配置 Web Server，实训环境友好 |
| 2 | 内存会话 (无 Redis) | 减少外部依赖，重启即失效可接受 |
| 3 | 原始 SQL (无 ORM) | C 语言不适合 ORM，直接 SQL 更可控 |
| 4 | 弱符号桩函数 | 允许模块逐步开发，`#if 1/#if 0` 控制 |
| 5 | Hash 路由 | 避免服务端配置 SPA fallback |
| 6 | 统一 HTTP 200 + code 字段 | 简化错误处理，前端统一判断 |
| 7 | Cookie 认证 | HttpOnly 防 XSS 窃取，比 localStorage Token 更安全 |
