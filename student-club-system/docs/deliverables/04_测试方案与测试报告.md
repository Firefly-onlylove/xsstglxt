### 三、接口测试用例

#### 3.1 认证模块

```bash
# 正常登录
curl -X POST http://127.0.0.1:8000/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}'
# 预期: {"code":0,"msg":"ok","data":{...}}

# 错误密码
curl -X POST http://127.0.0.1:8000/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"wrong"}'
# 预期: {"code":-8,"msg":"用户名或密码错误"}

# 获取当前用户
curl http://127.0.0.1:8000/api/me -b /tmp/cookies.txt
# 预期: {"code":0,"data":{"username":"admin","role":"school_admin"}}
```

### 六、历史 Bug 修复记录

| # | 问题 | 根因 | 修复 |
|---|------|------|------|
| 1 | 学院审批已通过但状态未更新 | college_reviewed 与 status 分离不彻底 | 引入二级审批链 |

| 7 | 重置密码后无法登录 | 默认密码前后端不一致 | 统一为 scms123456 |
| 8 | 社长端管理后台 404 | 路由表缺 dashboard/:id | 新增路由 |
| 9 | 社长端管理后台入口消失 | users.role=general_student 未同步 | /api/me 动态判定 role |
| 10 | 仪表盘待审批与财务管理页不一致 | pending 统计含已学院审批的 | 改为只统 college_reviewed='pending' |
| 11 | 收支记录列表为空 | SQL 查不存在的列 operator_name | LEFT JOIN users 取 real_name |
