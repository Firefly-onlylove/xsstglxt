## 一、项目背景

高校学生社团是校园文化的重要组成部分，但传统的人工管理模式面临诸多挑战：纸质申请流程繁琐、信息不对称、财务不透明、数据统计困难。本次实训旨在开发一套完整的 **学生社团管理系统 (SCMS)**，以 Web 应用的形式实现社团全生命周期的数字化、规范化管理。

系统采用 **C11 语言**编写后端（编译为单体可执行文件 `scms.exe`），**Vue 3 + Element Plus** 构建前端 SPA，**MySQL 8.0** 作为持久化存储。系统覆盖学校管理员、学院管理员、社团管理员、普通学生四类角色，提供社团审批、成员管理、活动组织、财务报销、换届选举、通知推送等完整业务流程。

## 二、设计思路

### 2.1 架构选型

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 后端语言 | C11 | 实训要求，考验 C 语言综合应用能力 |
| HTTP 框架 | Mongoose 嵌入式 | 单文件包含，零外部依赖，适合单体部署 |
| 数据库 | MySQL 8.0 + C API | 成熟稳定，C API 直接可用的关系型数据库 |
| 前端框架 | Vue 3 + Vite | 组件化开发效率高，生态完善 |
| UI 库 | Element Plus | 企业级组件，表格/表单/弹窗开箱即用 |
| 认证方案 | SHA256 + Cookie | 自行实现，不依赖 OpenSSL |

### 2.2 模块划分

后端按**角色维度**组织源码目录：

```
server/src/
├── common/     # 基础设施层 (DB/认证/API辅助/通知/工具)
├── school/     # 学校管理员业务 (9个模块)
├── college/    # 学院管理员业务 (6个模块)
└── student/    # 学生端 + 社团管理端 (14个模块)
```

前端按**路由区域**划分页面：

```
client/src/views/
├── school/      # 学校管理员页面 (9页)
├── college/     # 学院管理员页面 (6页)
├── student/     # 学生页面 (8页)
└── club-admin/  # 社团管理页面 (5页)
```

### 2.3 关键设计亮点

1. **二级审批链建模**：报销引入独立 `college_reviewed` 字段与 `status` 分离，实现学院初审→学校终审的完整审批链路
2. **动态角色判定**：`/api/me` 端点在返回用户角色时检查 `members` 表，自动将社长/副社长提升为 `club_admin`，避免 `users.role` 未同步导致权限丢失
3. **弱符号桩函数**：所有业务 handler 在 `handlers_stub.c` 中有弱符号默认实现，各模块通过 `#if 1` / `#if 0` 控制编译启用
4. **连接池 + 事务**：数据库层提供互斥锁保护的连接池和 `db_begin/commit/rollback` 事务接口，保证数据一致性

### 3.2 核心业务流程

**社团全生命周期：**
```
学生申请 → 学校审批(设定校级/院级) → 社长管理(成员/活动/财务/选举) → 可解散
```

**报销二级审批链：**
```
社长提交(status=pending, college_reviewed=pending)
  → 学院初审(college_reviewed=approved/rejected)
    → 学校终审(status=approved/rejected) → 自动记账 INSERT INTO finance
```

**活动状态机：**
```
草稿 → 已发布(可报名) → 进行中(可签到) → 已结束(可总结) → 已归档
  └→ 取消(任意状态) → 自动通知报名者
```

## 五、技术难点与解决方案

### 5.1 C 语言 Web 开发

**难点**：C 语言没有现代 Web 框架的路由、中间件、JSON 解析等基础设施，一切需从零构建。

**方案**：
- 基于 Mongoose 嵌入式 HTTP 库实现 URI 通配模式匹配
- 自建 JSON Builder (`JsonBuilder` 结构体)，通过 `json_add_*` 系列函数构造响应
- 实现 `api_get_*` 系列函数统一读取 query string / JSON body / path 参数
- 连接池使用互斥锁保护，事务接口封装 `db_begin/commit/rollback`

### 5.2 SQL 注入防护

**难点**：原始 SQL 拼接必须防范注入攻击。

**方案**：所有字符串参数在使用前调用 `db_escape()` (封装 `mysql_real_escape_string`)，数值参数使用 `%d`/`%.2f` 格式化（类型安全）。

### 5.3 前后端字段一致性

**难点**：C 后端手动拼接 JSON，与 Vue 前端 prop 名极易不一致。

**方案**：
- 统一命名规范：数据库字段用 snake_case，JSON 响应保持一致
- API 文档先行，curl 测试验证每个端点的返回字段
- 修复记录：`purpose` → `description`、`created_at` → `submitted_at`、`operator_name`(不存在的列) → LEFT JOIN users 取 `real_name`

### 5.4 MySQL affected_rows 语义

**难点**：`db_execute("UPDATE ... SET password='same_hash'")` 当新旧密码相同时，MySQL 返回 `affected_rows=0`，代码中 `if (ok <= 0)` 误判为"用户不存在"。

**方案**：UPDATE/DELETE 操作前先 `SELECT COUNT(*)` 确认记录存在，`db_execute` 返回值只用于判断 `ok < 0`（真正的数据库错误）。

### 5.5 二级审批链建模

**难点**：报销需要学院初审→学校终审两级，单字段无法区分"学院已批等学校"和"社团刚提交等学院"。

**方案**：引入独立的 `college_reviewed` ENUM 字段，与 `status` 分离：
- `status` = 报销最终状态 (pending/approved/rejected)
- `college_reviewed` = 学院审批状态 (pending/approved/rejected)

## 六、总结与反思

### 6.1 项目成果

- **代码规模**：后端 ~8000 行 C 代码 (31 个 .c 文件)，前端 ~5000 行 Vue 代码 (28 个页面)
- **数据库**：21 张表，5 个触发器，完整的 ER 建模
- **API 端点**：~100 个，覆盖四类角色的全部业务
- **测试通过率**：100%（已执行 47/47 测试用例）

### 6.2 团队协作反思

1. **沟通效率**：前后端字段名不一致是最频繁的问题 (purpose vs description, operator_name vs real_name)，后续项目应优先定义 API 契约文档
2. **代码评审**：协作者提交的代码引入过 `res.message` 而非 `res.msg`，以及 BOM 字符等问题，评审不充分导致合并后需要回退
3. **进度管理**：组长承担了 50% 的编码工作，任务分解粒度应在开发初期细化

### 6.3 技术成长

1. 深入理解了 C 语言在 Web 后端场景的应用边界和最佳实践
2. 掌握了 MySQL C API 的连接池管理、事务处理和字符集适配
3. 熟悉了 Vue 3 Composition API + Element Plus 的企业级前端开发模式
4. 积累了多角色权限系统、二级审批链、状态机等业务建模经验
5. 理解了 `mysql_affected_rows()` 在 UPDATE 值未变化时返回 0 的语义 —— 这不是"用户不存在"，而是"没有行被修改"

### 6.4 改进方向

1. 增加自动化测试脚本（curl 批量接口测试）
2. 引入 CI/CD (GitHub Actions 自动编译+测试)
3. 补全选举模块功能并完善测试
4. 增加文件上传大小限制和图片压缩
