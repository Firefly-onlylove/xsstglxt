## 一、引言

### 1.1 背景

高校学生社团数量众多，传统的纸质申请或分散的 Excel 管理模式已无法满足信息化管理需求。学生申请加入社团、社团组织活动、经费报销、换届选举等流程缺少统一平台支撑，学校管理层难以及时掌握全校社团动态。因此需要开发一套**学生社团管理系统 (SCMS)**，实现社团全生命周期的数字化管理。

### 1.2 目标

1. 为学校/学院管理员提供统一的管理视图（社团监控、用户管理、财务监督、数据统计）
2. 为社团社长提供自助管理工具（成员审批、活动发布、经费记录、选举组织）
3. 为学生提供便捷的社团浏览、加入、活动报名体验
4. 确保数据安全和权限隔离，不同角色只能访问授权范围内的数据

### 1.3 适用范围

本系统适用于高校场景，支持一所大学内多学院、多社团的管理。系统用户覆盖全校师生。

## 二、用户角色分析

### 2.1 角色定义

| 角色 | 角色值 | 描述 | 典型场景 |
|------|--------|------|---------|
| **学校管理员** | `school_admin` | 全校最高权限管理员 | 审批新建社团、管理全部用户、财务监督、系统配置 |
| **学院管理员** | `college_admin` | 分管各学院的管理员 | 管理本院班级、监督本院社团、审批本院报销 |
| **社团管理员** | `club_admin` | 社团社长/副社长 | 管理社团成员、发布活动、记录财务、组织选举 |
| **社团成员** | `club_member` | 已加入社团的学生 | 参加社团活动、参与选举投票 |
| **普通学生** | `general_student` | 未加入任何社团的学生 | 浏览社团广场、申请加入社团、报名活动 |

## 三、功能需求

### 3.1 公共功能 (无需登录)

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F-PUB-01 | 用户注册 | 学生填写用户名、密码、姓名、学号、学院、专业、班级、手机号注册 | P0 |
| F-PUB-02 | 用户登录 | 用户名 + 密码登录，下发 Session Token | P0 |
| F-PUB-03 | 获取个人信息 | GET /api/me 返回当前登录用户的角色、学院等信息 | P0 |
| F-PUB-04 | 学院/专业/班级查询 | 下拉选择器级联查询，方便注册时选择 | P1 |

## 四、非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| API 响应时间 | 普通请求 < 200ms，统计查询 < 2s |
| 并发用户数 | 支持 50 并发 |
| 数据库连接池 | 默认 5 连接，最大 16 |
| 前端首屏加载 | < 3s |

### 4.2 安全要求

| 指标 | 要求 |
|------|------|
| 密码存储 | SHA256 哈希，不可逆 |
| 会话管理 | 32 位随机 Token，HttpOnly Cookie |
| SQL 注入防护 | 所有用户输入 `mysql_real_escape_string` 转义 |
| 权限控制 | 路由级 need_login 校验 + 业务级 api_require_* 校验 |
| 文件上传 | 随机化文件名、扩展名白名单、目录穿越防护 |

### 4.3 可用性要求

| 指标 | 要求 |
|------|------|
| 浏览器兼容 | Chrome/Firefox/Edge 最新版 |
| 界面语言 | 中文 |
| 响应式 | PC 端优先，最小宽度 1200px |
| 错误提示 | 所有操作必须有成功/失败反馈 |

### 4.4 可维护性要求

- 模块化设计：后端按角色分目录(school/college/student)，前端按角色分路由
- 模块可选编译：`#if 1` / `#if 0` 控制，弱符号桩函数
- 代码注释：所有函数、结构体、枚举包含注释

## 五、业务规则与约束

### 5.1 社团管理规则

| 规则 | 说明 |
|------|------|
| R-CLB-01 | 社团级别分为校级和院级，校级社团全校可见，院级仅本学院可见 |
| R-CLB-02 | 每人可加入多个社团，但同一社团不得重复加入 |
| R-CLB-03 | 社长退出前须先转让社长职位 |
| R-CLB-04 | 学校管理员有权解散任何社团 |

### 5.2 活动管理规则

| 规则 | 说明 |
|------|------|
| R-ACT-01 | 活动状态流：草稿→已发布→进行中→已结束→已归档；任意状态可取消 |
| R-ACT-02 | 已发布状态可报名，进行中状态不可报名但可签到 |
| R-ACT-03 | 活动取消需填写原因，自动通知全体已报名者 |
| R-ACT-04 | 签到方式：6 位签到码或手动补签 |

### 5.3 财务规则

| 规则 | 说明 |
|------|------|
| R-FIN-01 | 报销审批链：社长提交 → 学院审批(college_reviewed) → 学校终审(status) |
| R-FIN-02 | 院级社团需经学院+学校两级审批，校级社团直接学校审批 |
| R-FIN-03 | 报销需上传发票图片，单次和周期累计额度限制 |
| R-FIN-04 | 学校终审通过后自动插入 finance 支出记录 |

### 5.4 选举规则

| 规则 | 说明 |
|------|------|
| R-ELE-01 | 选举分报名期和投票期两个阶段 |
| R-ELE-02 | 投票为匿名投票，每成员每职位限投 1 票 |
| R-ELE-03 | 平票时需重新投票或由学院管理员裁决 |

## 六、数据需求

### 6.1 数据量估算

| 数据实体 | 预估量级 | 说明 |
|---------|---------|------|
| 用户 | ~5000 | 含全部学生 + 管理员 |
| 社团 | ~80 | 校级约 20 个 + 院级每学院约 10 个 |
| 成员关系 | ~5000 | 每社团平均 60 人 |
| 活动 | ~200/年 | 每社团年均 2-3 个 |
| 财务流水 | ~1000/年 | |
| 通知 | ~10000 | 含系统自动通知 |

### 6.2 数据保留策略

- 操作日志：保留 1 年
- 通知消息：保留 6 个月
- 已解散社团：标记 `status=dissolved`，数据不物理删除

## 八、验收标准

1. 四类角色 (学校管理员/学院管理员/社团管理员/学生) 均能登录并完成核心流程
2. 全部 API 端点 (约 100 个) 返回正确的 JSON 格式
3. 社团从申请创建到审批通过的完整流程走通
4. 报销从提交到学院审批到学校终审的全流程走通（含财务记账联动）
5. 活动从创建到发布到报名到签到到结束的全生命周期正常
6. 权限隔离：用户无法访问非授权范围的页面和数据
7. 所有输入校验生效，SQL 注入和 XSS 防护到位
