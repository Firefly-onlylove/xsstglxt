## 一、引言

### 1.1 项目背景

高校学生社团是校园文化的重要载体，但随着社团数量增加和活动日益丰富，传统的人工管理模式暴露出诸多问题：纸质申请审批流程繁琐、社团信息分散难以查询、活动报名统计依赖人工、财务收支缺乏透明记录、换届选举缺少规范化流程。本次实训项目旨在开发一套完整的**学生社团管理系统（SCMS）**，以 Web 应用的形式实现社团全生命周期的数字化管理。

### 1.2 项目目标

- 实现社团从创建、运营到解散的全生命周期数字化管理
- 覆盖四类角色（学校管理员、学院管理员、社团管理员、学生）的完整业务场景
- 提供社团活动发布、报名、签到的完整流程
- 实现财务收支记录与报销二级审批（学院初审→学校终审）
- 支持数据统计与可视化展示
- 满足工程化开发要求：模块化、规范化、健壮性

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| 学校管理员 | 校级系统管理员，拥有最高管理权限 |
| 学院管理员 | 各学院的管理人员，负责本学院社团和学生的监管 |
| 社团管理员 | 社团的社长或副社长，负责本社团日常管理 |
| 普通学生 | 系统主要使用者，可浏览社团、报名活动 |
| 社团级别 | 社团分为校级社团和院级社团 |
| 二级审批 | 报销需经过学院初审和学校终审两级审批 |

## 二、功能需求

### 2.1 用户认证模块

#### 2.1.1 登录

- **FR-LOGIN-001**：用户通过用户名和密码登录系统
- **FR-LOGIN-002**：登录成功后下发会话 Cookie（SHA256 哈希 token），后续请求自动识别身份
- **FR-LOGIN-003**：登录失败时给出明确错误提示（用户名不存在 / 密码错误）
- **FR-LOGIN-004**：前端根据登录返回的角色自动跳转到对应首页

#### 2.1.2 注册

- **FR-REG-001**：学生可自助注册账号，需填写用户名、姓名、密码、学号、学院、专业、班级、手机号
- **FR-REG-002**：学号校验为 10 位数字，手机号校验为 11 位
- **FR-REG-003**：密码长度不少于 6 位
- **FR-REG-004**：用户名唯一性校验，重复时提示错误

#### 2.1.3 会话管理

- **FR-SESSION-001**：通过 Cookie 传递会话 token，HttpOnly 防 XSS 窃取
- **FR-SESSION-002**：`/api/me` 端点返回当前用户信息，前端路由守卫据此做权限校验
- **FR-SESSION-003**：动态角色判定：用户在 members 表中是社长/副社长时自动获得 club_admin 角色
- **FR-SESSION-004**：登出时清除服务端会话和客户端 Cookie

### 2.2 学校管理员功能

#### 2.2.1 仪表盘

- **FR-SCH-001**：展示全校社团总数、学生总数、活动总数核心统计
- **FR-SCH-002**：展示待审批社团数量、待审批报销数量
- **FR-SCH-003**：展示全校社团总经费（本学期收支汇总）

#### 2.2.2 社团审批

- **FR-SCH-004**：查看学生提交的创建社团申请列表
- **FR-SCH-005**：审批通过时设定社团级别（校级/院级）、指定归属学院
- **FR-SCH-006**：审批拒绝时填写拒绝理由，通知申请人
- **FR-SCH-007**：已审批的申请保留在列表中，可查看审批结果

#### 2.2.3 社团管理

- **FR-SCH-008**：查看全部社团列表，支持按名称搜索、按状态/级别筛选
- **FR-SCH-009**：修改社团级别（校级↔院级）、归属学院
- **FR-SCH-010**：解散社团（设置社团为已注销状态）

#### 2.2.4 用户管理

- **FR-SCH-011**：查看全部用户列表，支持按姓名/学号/学院筛选
- **FR-SCH-012**：新增用户（管理员手动创建）
- **FR-SCH-013**：重置用户密码
- **FR-SCH-014**：限制/解除限制用户登录

#### 2.2.5 财务监督

- **FR-SCH-015**：查看各学院财务收支汇总
- **FR-SCH-016**：查看全部报销申请列表，区分待审批、已审批、已拒绝
- **FR-SCH-017**：对学院已批的报销进行学校终审（通过/驳回）
- **FR-SCH-018**：审批通过后自动记入社团财务支出记录

#### 2.2.6 统计中心

- **FR-SCH-019**：按学院统计社团数量分布
- **FR-SCH-020**：按月统计社团创建趋势
- **FR-SCH-021**：展示活动数量、参与人数等核心运营数据

#### 2.2.7 其它

- **FR-SCH-022**：发布全校公告，支持编辑和删除
- **FR-SCH-023**：系统配置管理（学期时间等）
- **FR-SCH-024**：数据备份功能

### 2.4 社团管理员功能

## 三、非功能需求

### 3.1 性能需求

- **NFR-001**：数据库连接池最大 16 连接，默认 5 连接，支持并发请求
- **NFR-002**：列表接口均支持分页（默认每页 10 条），避免大数据量查询
- **NFR-003**：静态资源使用 Vite 打包压缩，前端首屏加载时间合理

### 3.2 安全需求

- **NFR-004**：密码使用 SHA256 哈希存储，不明文保存
- **NFR-005**：所有用户输入经 `mysql_real_escape_string` 转义防 SQL 注入
- **NFR-006**：数值参数使用 `%d` 格式化，类型安全防注入
- **NFR-007**：Session token 为 32 位随机十六进制字符串，防伪造
- **NFR-008**：Cookie 设置 HttpOnly 标志，防 XSS 窃取
- **NFR-009**：文件上传自动重命名为时间戳+随机码，防路径穿越和覆盖
- **NFR-010**：权限校验多层防御：路由守卫 + API 权限函数 + 数据归属校验

### 3.3 数据一致性

- **NFR-011**：社团成员计数通过触发器自动维护（INSERT/UPDATE/DELETE 自动同步）
- **NFR-012**：活动报名计数通过触发器自动维护
- **NFR-013**：报销审批通过后自动插入 finance 表记录（事务保证）
- **NFR-014**：成员退出所有社团后自动将 users.role 降回 general_student

### 3.4 可用性

- **NFR-015**：所有 API 错误返回明确的中文错误提示
- **NFR-016**：前端操作成功后自动刷新关联数据列表
- **NFR-017**：未登录访问 API 返回明确提示，前端自动跳转登录页

### 3.5 可维护性

- **NFR-018**：后端按角色维度组织源码目录（common/school/college/student/）
- **NFR-019**：所有业务函数在 handlers_stub.c 中有弱符号默认实现
- **NFR-020**：模块通过 `#if 1` / `#if 0` 控制编译启用，互不干扰

## 五、用例分析

### 5.1 核心用例：社团创建审批流程

```
参与者：学生、学校管理员

前置条件：学生已登录

主流程：
1. 学生提交创建社团申请（名称、简介、理由、Logo）
2. 学校管理员在审批列表看到申请
3. 学校管理员审核通过，设定社团级别（校级/院级）和归属学院
4. 社团创建成功，申请人自动成为社长
5. 系统通知申请人审批结果

备选流程：
3a. 学校管理员拒绝申请 → 填写拒绝理由 → 通知申请人
```

### 5.2 核心用例：报销二级审批流程

```
参与者：社团管理员、学院管理员、学校管理员

前置条件：社团已存在，申请人有管理权限

主流程：
1. 社团管理员提交报销申请（金额、描述、发票图片）
2. 学院管理员在审批列表看到申请（仅院级社团的报销）
3. 学院管理员初审通过（college_reviewed = approved）
4. 学校管理员在审批列表看到学院已批的报销
5. 学校管理员终审通过（status = approved）
6. 系统自动向 finance 表插入一条支出记录（type = expense）
7. 社团财务余额自动更新

备选流程：
3a. 学院管理员驳回 → status = rejected → 通知社团
5a. 学校管理员驳回 → status = rejected → 通知社团
```

### 5.3 核心用例：活动生命周期

```
参与者：社团管理员、学生

主流程：
1. 社团管理员创建活动（草稿状态）
2. 发布活动（已发布状态，学生可见可报名）
3. 活动开始，管理员标记进行中（可签到）
4. 活动结束，管理员标记已结束
5. 管理员归档活动（已归档）

备选流程：
2a. 管理员取消活动 → 自动通知已报名学生
```

## 六、数据需求

### 6.1 核心数据实体

| 实体 | 核心属性 | 说明 |
|------|---------|------|
| 用户 (users) | user_id, username, password_hash, real_name, student_no, role, college_id, class_id | 核心用户表 |
| 社团 (clubs) | club_id, club_name, level, college_id, status, member_count | 社团信息表 |
| 成员 (members) | member_id, club_id, user_id, role, join_status, joined_at, left_at | 社团成员关系表 |
| 活动 (activities) | activity_id, club_id, title, status, start_time, end_time, max_participants | 活动信息表 |
| 活动报名 (activity_registrations) | reg_id, activity_id, user_id, signin_status | 活动报名与签到 |
| 财务记录 (finance) | finance_id, club_id, type, amount, description, source, operator_id, record_time | 社团财务 |
| 报销申请 (reimbursement) | reimb_id, club_id, amount, description, receipt_path, status, college_reviewed, submitted_at | 报销审批 |
| 选举 (elections) | election_id, club_id, title, status, nomination_deadline, vote_deadline | 换届选举 |
| 通知 (notifications) | notif_id, user_id, title, content, type, is_read | 消息通知 |
| 公告 (announcements) | ann_id, target_type, target_id, title, content, publisher_id | 各类公告 |

### 6.2 数据量估算

| 实体 | 预估行数 | 说明 |
|------|---------|------|
| users | ~500 | 全校学生用户 |
| clubs | ~30 | 全校社团 |
| members | ~500 | 每社团 10-50 人 |
| activities | ~200/学期 | 每月 40 场活动 |
| finance | ~500/学期 | 每次活动和报销产生 |
