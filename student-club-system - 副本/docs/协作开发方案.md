## 二、分支策略（强烈建议采用）

```
main              ← 稳定发布版（仅合并，禁止直接提交）
 └── dev          ← 日常开发集成分支
      ├── fix/xxx ← bug 修复分支
      └── feat/xxx ← 新功能分支
```

### 当前问题

你们目前共用分支 `1`，两人直接向同一分支推送。这会导致：
- `git push` 被拒绝时需要 `git pull --rebase` 再推，容易产生无意义的合并提交
- 一个人提交了半成品，另一个人拉下来编译不过
- 无法 Code Review

### 标准工作流（单人每天）

```bash
# === 早上第一件事 ===
git checkout dev
git pull origin dev

# === 开始一个新任务 ===
git checkout -b fix/something-broke    # 修 bug
git checkout -b feat/new-feature       # 新功能

# === 写代码... ===
git add <只加源文件>
git commit -m "fix: 修复 xxx 问题"

# === 推送到远程 ===
git push origin fix/something-broke

# === 合并到 dev ===
git checkout dev
git pull origin dev          # 先拉取别人可能的新提交
git merge fix/something-broke
git push origin dev

# === 删除已合并的特性分支 ===
git branch -d fix/something-broke
git push origin --delete fix/something-broke
```

### 如果继续共用分支 `1`（简化版）

```bash
# 每天早上同步
git pull origin 1 --rebase

# 写代码后提交
git add <文件>
git commit -m "类型: 描述"

# 推送前再次同步
git pull origin 1 --rebase
git push origin 1
```

> **关键规则：推送前必须先 pull --rebase，保持历史线性。**

## 三、提交规范

### 格式

```
<type>: <简短描述>

<详细说明（可选）>
```

### type 类型

| 前缀 | 用途 | 示例 |
|------|------|------|
| `fix:` | 修 bug | `fix: 修复禁用用户后无提示` |
| `feat:` | 新功能 | `feat: 社团活动报名接口` |
| `chore:` | 构建/配置 | `chore: 更新 .gitignore` |
| `docs:` | 文档 | `docs: 添加协作开发规范` |
| `refactor:` | 重构 | `refactor: 提取公共校验函数` |
| `style:` | 样式调整 | `style: 修复表格对齐` |

### 规则

- 标题不超过 72 个字符
- 提交粒度：**一个 commit 只做一件事**
- 不要提交 `public/` 目录（构建产物）
- 不要提交 `db_config.ini`（含密码）
- 不要提交 `node_modules/`

## 四、冲突预防清单

### 文件级

| 规则 | 原因 |
|------|------|
| `public/` 必须 `.gitignore` | 每台设备各自 `npm run build`，提交会产生冲突 |
| `db_config.ini` 必须 `.gitignore` | 两台设备 MySQL 密码不同 |
| `node_modules/` 不提交 | 用 `npm install` 恢复 |
| `scms.exe` 不提交 | 每台设备各自编译 |

### 操作级

- [ ] 每天开始工作前：`git pull origin 1 --rebase`
- [ ] 每次提交前：`git status` 确认文件清单
- [ ] 推送前：再拉一次，确保没有新冲突
- [ ] 改了数据库结构：同步更新 `database/02_create_tables.sql` 并告知对方
- [ ] 新增依赖：告知对方执行 `npm install`

## 五、数据库同步

两台设备各自运行本地 MySQL 8.0，数据互不影响。

### 结构变更流程

1. 修改表结构后，更新 `database/02_create_tables.sql`
2. 在 commit message 中标注 `[DB]`
3. 通知对方重建数据库

### 对方更新后

```bash
# 方式一：重置数据库（数据会丢失）
mysql -u root -p123456789 -e "DROP DATABASE IF EXISTS scms_db; CREATE DATABASE scms_db DEFAULT CHARSET utf8mb4;"
mysql -u root -p123456789 scms_db < database/02_create_tables.sql
mysql -u root -p123456789 scms_db < database/03_seed_data.sql
# ... 依序导入

# 方式二：手动对比差异执行 ALTER TABLE（保留数据，推荐生产环境）
```

### 数据库信息

| 项目 | 值 |
|------|-----|
| 数据库名 | `scms_db` |
| 用户 | `root` / `scms_user` |
| 字符集 | `utf8mb4` |
| SQL 目录 | `database/` |

## 七、冲突处理流程

### 当 git pull 出现冲突时

```bash
# 1. 查看冲突文件
git status

# 2. 手动编辑冲突文件，找到 <<<<<<< ======= >>>>>>> 标记
#    保留正确的内容，删除标记行

# 3. 标记已解决
git add <已解决的文件>

# 4. 继续 rebase
git rebase --continue

# 5. 推送
git push origin 1
```

### 如果冲突太复杂

```bash
# 放弃本次 rebase，回到之前状态
git rebase --abort

# 换用 merge 方式
git pull origin 1
# 手动解决冲突后提交
```

## 九、常见问题速查

| 问题 | 解决方法 |
|------|----------|
| push 被拒绝 (non-fast-forward) | `git pull origin 1 --rebase` 后再 push |
| 编译不过 (undefined reference) | 检查 `MYSQL_INC` / `MYSQL_LIB` 路径，见 `server/Makefile` |
| 前端白屏 | 运行 `cd client && npm run build` 重新构建 |
| 数据库连接失败 | 检查 `db_config.ini` 是否存在且密码正确 |
| ngrok 显示 Bad Gateway | 先确保 scms.exe 正在运行，再启动 ngrok |

## 十、协作沟通规则

1. **改数据库结构 → 微信群通知对方**，附上 SQL 变更内容
2. **改了公共模块**（`common/`、`main.c`、`routes.c`）→ 告知对方
3. **提交前自查**：`git diff --stat` 确认只改了该改的文件
4. **遇到冲突不要慌**：先看冲突文件列表，能解决自己解决，不能解决截图发群里
5. **定期同步**：至少每天一次 `git pull`
