## 一、项目背景

高校学生社团是校园文化的重要组成部分。随着社团种类和数量的增长，传统的人工管理模式——纸质申请表、口头协调、手工记账——已无法满足高效、规范、透明的管理需求。

本次实训第二阶段为企业级团队实战项目，要求 4-6 人团队协作，模拟完整的软件开发流程，从需求分析、系统设计到编码实现、测试验收，全面考察 C 语言综合应用能力、团队协作能力与工程实践能力。我们选择开发**学生社团管理系统（SCMS）**，以 Web 应用的形式，为学校管理员、学院管理员、社团管理员和学生四类角色提供一站式的社团管理平台。

系统覆盖社团全生命周期——从学生申请创建社团、学校审批、社长运营管理（成员审批任命、活动发布签到、财务收支报销、换届选举）、到社团解散的完整业务闭环。技术上采用 C11 + MySQL 8.0 作为后端，Vue 3 + Element Plus 作为前端，前后端分离架构，通过 JSON API 通信。

## 二、设计思路

### 2.1 为什么用 C 语言写后端

实训核心要求是 C 语言全知识点的综合应用。选择用 C 写 Web 后端是一项有挑战性的决策——C 没有现成的路由框架、没有中间件、没有 JSON 解析库、没有 ORM，一切基础设施都需要从零构建。这恰好体现了工程实践能力的综合考察：

- **指针与内存管理**：JsonBuilder 动态缓冲区、db_escape 转义内存分配、MYSQL_RES 结果集处理，每处 malloc 都对应明确的 free 时机
- **结构体与数据建模**：ApiContext 请求上下文、User/Club 数据模型、Route 路由表结构体，体现用结构体组织复杂数据的能力
- **函数指针**：路由表中存储 `void (*handler)(ApiContext*)` 函数指针，实现灵活的业务分发
- **预处理器宏**：`MG_ESC` 安全字符串输出、`#if 1`/`#if 0` 模块编译开关、ERR\_\* 系列错误码
- **文件操作**：`db_config.ini` 配置读取、文件上传保存
- **位运算与算法**：自行实现 SHA256 哈希（200+ 行纯 C：消息填充、64 轮哈希计算、十六进制输出）

### 2.2 架构选型

| 决策点    | 选择                   | 核心理由                                    |
| --------- | ---------------------- | ------------------------------------------- |
| 后端语言  | C11                    | 实训核心要求                                |
| HTTP 框架 | Mongoose（单文件嵌入） | 极轻量，无需外部依赖，直接 `#include` 编译  |
| 数据库    | MySQL 8.0 + 原生 C API | C 语言直接调用，无 ORM 抽象层，完全控制 SQL |
| 前端框架  | Vue 3 Composition API  | 组件化开发，与 Element Plus 无缝集成        |
| UI 组件   | Element Plus 2.7       | 开箱即用的企业级表格/表单/弹窗组件          |
| 路由      | Vue Router 4 Hash 模式 | 无需服务端配置，部署简单                    |
| 认证      | SHA256 + Cookie Token  | 不依赖 OpenSSL，完全自实现                  |

### 2.3 目录结构与模块划分

后端按角色维度组织源码，按业务聚合，而非按技术层：

```
server/src/
├── common/         — 基础设施（db, auth, api, routes, utils, notification）
├── school/         — 学校管理员业务（9 个 .c 文件）
├── college/        — 学院管理员业务（6 个 .c 文件）
└── student/        — 学生端（8 个）+ 社团管理端（7 个，club_admin/ 子目录）
```

这种组织方式的优势是：每个模块只负责对应角色的业务逻辑，接口职责单一，团队成员互不干扰。

### 2.4 关键设计

**弱符号桩函数机制**：所有业务处理函数在 `handlers_stub.c` 中有弱符号默认实现（返回"功能尚未实现"），各模块通过 `#if 1` / `#if 0` 控制编译。这种方式让模块可以逐个启用，未完成的模块自动降级为占位，不会阻塞其他模块。

**连接池 + 事务**：数据库层提供互斥锁保护的连接池（默认 5 连接，最多 16），连接用 `pthread_mutex_t` 保护，`db_begin/commit/rollback` 封装事务，确保报销审批→自动记账的原子性。

**动态角色判定**：`/api/me` 端点返回角色时，不直接读 `users.role` 字段，而是额外检查 `members` 表——若用户在任一社团担任 president/vice_president，动态返回 `club_admin` 角色。这解决了 `users.role` 未及时更新的问题，是防御性设计。

**二级审批链建模**：报销表引入独立的 `college_reviewed` ENUM 字段，与 `status` 解耦。`status` 表示最终审批状态，`college_reviewed` 表示学院环节状态。三级审批可以在同一模型上扩展。


## 五、技术难点与解决方案

### 5.1 C 语言 Web 基础设施从零构建

**难点**：C 语言没有现代 Web 框架的路由分发、中间件链、JSON 解析、请求参数解析等基础设施。

**方案**：

- 利用 Mongoose 的 URI 模式匹配实现通配路由（`*` 匹配单一路径段）
- 自建 `ApiContext` 结构体统一封装连接、请求消息、用户信息
- `api_get_param` 统一读取 query string 和 POST body；`api_get_path_int` 从 URL 提取路径参数；`api_get_json_str` 解析 JSON body
- `JsonBuilder` 结构体以 `realloc` 动态扩容方式逐字段追加 JSON，避免固定缓冲区溢出
- `api_send_result` 自动将 MySQL 结果集转为 JSON 数组，根据列类型智能判断字段是否加引号

### 5.2 SHA256 密码哈希的自实现

**难点**：不依赖 OpenSSL 等外部库，需从头实现 SHA256 算法。

**方案**：实现完整的 SHA256 流程——

- 消息预处理：原始消息追加 `0x80` 字节，填充至 448 mod 512，末尾写入 64 位消息长度
- 每 512 位块进行 64 轮哈希运算，包含 Message Schedule 扩展（W[0..15] 直接取自块，W[16..63] 由前值通过 σ0 和 σ1 函数计算）
- 使用预设的 8 个初始哈希值 H[0..7] 和 64 个轮常量 K[0..63]
- 最终输出为 256 位（64 位十六进制字符）

### 5.3 SQL 注入防护的双层体系

**难点**：原始 SQL 拼接必须防范注入攻击，这是后端安全的首要问题。

**方案**：构建双层防护——

- **第一层（核心）**：所有字符串值在拼入 SQL 前经过 `db_escape()` 转义，内部调用 `mysql_real_escape_string()`，转义单引号、反斜杠、空字符等特殊字符。转义后的内存立即 `free()` 避免泄漏
- **第二层（类型安全）**：数值参数使用 `%d` / `%.2f` / `%ld` 格式化，编译器保证类型，无法注入非数字内容
- **约定禁止**：不使用裸字符串拼接，所有 SQL 通过 `db_query(fmt, ...)` 格式化构造

### 5.4 MySQL affected_rows = 0 的语义陷阱

**难点**：`db_execute("UPDATE ... SET password='same_hash'")` 当新旧密码相同时，MySQL 返回 `affected_rows=0`。代码中 `if (ok <= 0)` 将其误判为"用户不存在"，导致修改密码时报错。

**方案**：修改密码等 UPDATE/DELETE 操作前，先 `SELECT COUNT(*)` 确认记录存在，`db_execute` 返回值仅用于判断 `ok < 0`（真正的数据库执行错误）。

### 5.5 前后端字段名一致性

**难点**：C 后端手动拼接 JSON 字段名（如 `"description"`），Vue 前端模板中写 `prop: 'description'`，两处独立维护极易不一致。

**实际遇到的不一致问题**：

- 前端 `prop: 'purpose'` vs 后端 JSON `"description"` → 列不显示
- 前端 `prop: 'created_at'` vs 后端 JSON `"submitted_at"` → 时间列为空
- 后端 SQL 写 `f.operator_name`（列不存在）→ 前端始终为空

**方案**：API 开发完成后先用 curl 查看实际返回字段，前端 `prop` 严格对齐。数据库列名 = JSON 字段名 = 前端 prop 名，全链路用 snake_case。

### 5.6 member_id 与 user_id 的区分

**难点**：`members` 表中 `member_id` 是主键（自增，从 1 开始），`user_id` 是外键指向 `users` 表。API 路由 `/api/club/{id}/members/{mid}/appoint` 中的 `{mid}` 取的是 `member_id`，但前端传入 `row.user_id`（如 205），导致查询 `member_id=205` 找不到记录。

**方案**：前端三个操作函数（setManager/demoteToMember/removeMember）统一改用 `row.member_id`，与后端 `api_get_path_int(ctx, 3)` 取 `mid` 对齐。member_id 从 API 返回的 `member_id` 字段取，不是 `user_id`。

### 5.7 数据库连接池的多线程安全

**难点**：服务端每收到一个 HTTP 请求，Mongoose 在当前线程中回调事件处理函数。多个并发请求同时访问数据库时，如果共享同一个 `MYSQL*` 连接，会出现竞态条件。

**方案**：

- 维护 `MYSQL*` 连接池数组 + `pthread_mutex_t` 锁数组
- `db_get_conn()` 遍历连接池，找到未锁定连接后上锁并返回
- `db_release_conn()` 解锁归还
- 连接池大小可配置（`db_config.ini` 中 `pool_size=5`），最大 16

## 七、总结与反思

### 7.1 项目收获

1. **C 语言系统编程能力**：完整经历了用 C 从零构建 Web 服务器的全过程——HTTP 协议处理、路由分发、数据库连接池、认证会话、JSON 序列化，对 C 语言的指针、内存管理、系统调用有了深层次理解。

2. **数据库设计能力**：从 ER 图到 22 张表的 DDL，从索引设计到触发器自动维护计数，完整掌握了关系型数据库在业务系统中的建模方法。

3. **企业级软件开发流程**：实践了需求分析→系统设计→编码实现→测试验证→文档交付的完整流程，理解了每个阶段的交付物和验收标准。

4. **工程化安全意识**：SQL 注入防护、密码哈希存储、文件上传安全、权限多层校验——这些在真实项目中至关重要的安全问题，在本次实训中得到了实践。

5. **团队协作经验**：Git 分支管理、代码评审、每日站会、进度跟踪——这些企业级协作实践让我们理解了"写代码"和"做项目"的本质区别。

### 7.2 团队协作反思

1. **沟通效率是关键**：前后端字段名不一致是最高频的联调问题（purpose vs description, operator_name vs real_name, created_at vs submitted_at），如果前期花一小时约定好 API 契约文档，可以节省数小时的联调时间。

2. **代码评审不可缺失**：协作者提交的代码出现过 `res.message`（应为 `res.msg`）、BOM 隐藏字符、catch 块静默吞错误等问题。评审不充分导致这些隐蔽问题合并后才被发现，后续采取了更严格的评审流程。

3. **任务粒度要适中**："完成学院端后端 6 个模块"这样的任务过于粗放，应该拆分为"第 1 天完成 college_dashboard.c 并自测通过"这样可验收的细粒度任务。

### 7.3 改进方向

1. 增加自动化接口测试脚本，避免手动 curl 逐个验证端点
2. 引入 CI/CD（GitHub Actions 自动编译 C 代码、构建前端、运行测试）
3. 日志系统升级为结构化日志（当前用 logs 表），支持日志级别和文件输出
4. 完善 WebSocket 实时通知（当前消息需要手动刷新页面）
5. 增加 API 限流机制，防止恶意高频请求
