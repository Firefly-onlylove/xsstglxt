## 二、协作规范

### 2.1 代码管理

- **分支策略**：`1`（主分支）+ `feature/*`（功能分支）
- **提交规范**：`<type>: <简短描述>`，type ∈ feat / fix / docs / refactor / test
- **合并流程**：功能分支 → PR → 组长代码评审 → 合并至主分支
- **代码风格**：C 代码遵循 K&R 风格，Vue 代码遵循 Vue 3 官方风格指南

### 2.2 沟通机制

| 机制     | 频率           | 内容                         |
| -------- | -------------- | ---------------------------- |
| 每日站会 | 每天 9:30      | 昨日进度、今日计划、阻塞问题 |
| 进度同步 | 每天 18:00     | 当日完成情况简报             |
| 代码评审 | 每次 PR 合并前 | 组长主审，关注安全与正确性   |

### 2.3 文档规范

- 所有文档使用 Markdown 格式，版本号格式 `v<major>.<minor>`
- 流程图使用 Mermaid 或 ASCII Art
- API 字段名统一使用 snake_case，与数据库字段保持一致

## 三、风险管理

| 风险                                 | 概率 | 影响 | 应对措施                                           |
| ------------------------------------ | ---- | ---- | -------------------------------------------------- |
| MySQL C API 编译问题（MinGW 兼容性） | 高   | 高   | 第 1 周提前验证编译链，准备静态链接方案            |
| C 语言内存管理（无 GC、无异常）      | 中   | 高   | 统一 malloc/free 配对规范，db_escape 用后立即 free |
| 前后端字段名不一致                   | 高   | 中   | API 文档先行，curl 验证每个端点返回字段后再写前端  |
| MySQL affected_rows 语义问题         | 中   | 中   | UPDATE 前先 SELECT 确认记录存在，避免 0 行误判     |
| 数据库字符集乱码                     | 中   | 中   | 统一 utf8mb4，每次连接 SET NAMES utf8mb4           |
| 进度延期                             | 中   | 高   | 组长持有弹性调配权，日报机制及时发现问题           |
| 公网部署访问问题                     | 中   | 中   | 确保 CORS 头完整、Cookie SameSite 配置正确         |
