## 第一阶段：基础设施层（必须精通）

### 1.1 数据库层 `server/src/common/db.c` + `include/db.h`

#### db.c 全局结构
整个文件用一个全局结构体 `g_db` 管理所有状态：
```c
static struct {
    DBConfig  cfg;                    // 数据库配置（host/port/user/password/database/pool_size）
    PoolSlot  slots[MAX_POOL];       // 连接池槽位数组（最多16个）
    int       size;                   // 实际池大小（默认5）
    db_mutex_t lock;                  // 互斥锁（Windows: CRITICAL_SECTION, Linux: pthread_mutex_t）
    int       initialized;            // 是否已初始化
    char      last_error[512];        // 最近一次错误信息
} g_db = {0};
```

- [ ] **连接池机制 `PoolSlot`**：每个槽位包含 `MYSQL *conn`（连接句柄）和 `int in_use`（占用标记）。池大小默认 5、最大 16。互斥锁保护所有获取/释放操作，保证多线程安全（虽然当前 Mongoose 是单线程事件循环，但为将来扩展考虑加了锁）。

- [ ] **`create_connection()`**（第56-73行）：建立单条 MySQL 连接的完整流程：
  1. `mysql_init(NULL)` 创建连接句柄
  2. `mysql_options(conn, MYSQL_SET_CHARSET_NAME, "utf8mb4")` 设置字符集（**防止中文乱码的关键步骤**）
  3. `mysql_real_connect()` 用配置中的 host/user/password/database/port 连接
  4. `mysql_set_character_set(conn, "utf8mb4")` 二次确认字符集
  5. 任一步失败 → 记录错误信息到 `g_db.last_error` → `mysql_close()` → 返回 NULL

- [ ] **`db_init(const DBConfig *cfg)`**（第75-106行）：
  - 校验 `pool_size`：≤0 则默认 5，>16 则截断为 16
  - `MUTEX_INIT(&g_db.lock)` 初始化互斥锁
  - `mysql_library_init()` MySQL 库全局初始化
  - 循环调用 `create_connection()` 填充每个池槽位
  - **全部连接都建立成功才返回 OK**，有一条失败则关闭已建立的并返回 ERR_DB（全有或全无）

- [ ] **`db_get_conn()`**（第123-138行）：
  - 加锁 → 遍历槽位找第一个 `in_use == 0` 的槽 → 标记 `in_use = 1` → 解锁 → 返回连接
  - **池满时的降级策略**：如果所有槽位都被占用，临时新建一条连接（不放入池），用完由 `db_release_conn` 判断并关闭

- [ ] **`db_release_conn(MYSQL *conn)`**（第140-153行）：
  - 加锁 → 在池中查找该连接 → 找到则标记 `in_use = 0` → 解锁
  - **找不到则该连接是池满时临时创建的** → 直接 `mysql_close()` 关闭

- [ ] **`format_sql()`**（第156-164行）：
  - 用 `vsnprintf()` 将 printf 风格的格式串 + 变参拼成最终 SQL
  - SQL_BUF_SIZE = 8192 字节
  - 超长则记录错误，返回 -1

- [ ] **`db_query(const char *fmt, ...)`**（第188-213行）：
  - 参数 → `format_sql` → `db_get_conn` → `mysql_query(conn, sql)` → `mysql_store_result(conn)` → `db_release_conn` → 返回 `MYSQL_RES*`
  - `mysql_store_result` 返回 NULL 且无错误码 = 该语句无结果集（正常情况，如纯 INSERT）
  - **调用方负责对返回值调用 `mysql_free_result()`**

- [ ] **`db_execute(const char *fmt, ...)`**（第166-186行）：
  - 与 `db_query` 流程相同，但不调用 `mysql_store_result`，而是调用 `mysql_affected_rows(conn)` 获取受影响行数
  - **返回值**：受影响行数（≥0），失败返回 -1
  - **使用场景**：INSERT/UPDATE/DELETE 等无结果集的 SQL

- [ ] **`db_last_insert_id()`**（第215-221行）：
  - 获取最近一次 INSERT 操作的自增 ID
  - 内部获取一个池连接 → `mysql_insert_id(conn)` → 释放 → 返回 ID
  - 用于插入记录后获取新记录的 ID

- [ ] **事务支持**（第228-238行）：
  - `db_begin()` → `START TRANSACTION`
  - `db_commit()` → `COMMIT`
  - `db_rollback()` → `ROLLBACK`
  - **注意**：这是简化实现，通过开关 autocommit 实现。注释中提到如需严格事务建议在单次请求内串行完成

- [ ] **`db_escape(const char *str)`**（第240-258行）：
  - 用 `mysql_real_escape_string()` 转义字符串中的特殊字符（引号、反斜杠等）
  - **返回值是 `malloc` 分配的**，用完必须 `free()`
  - 分配大小 = `原始长度 × 2 + 1`（最坏情况每个字符都需要转义）
  - NULL 输入 → 返回空串 `""`

- [ ] **便捷取值函数**（第278-333行）：
  - `db_query_int(fmt, ...)`：执行查询 → 取第一行第一列 → `atoi()` 转整数 → 返回。无结果返回 0
  - `db_query_double(fmt, ...)`：同上，用 `atof()` 转浮点数
  - `db_query_str(fmt, char *out, size_t out_size, ...)`：取第一行第一列字符串到 out 缓冲区，无结果置空串。**注意参数顺序**：fmt 在前，out/out_size 在变参之前（但 C 的 va_arg 机制要求它们在 fmt 后面...实际上代码里是通过格式化的 SQL 中用 `%s` 拼入的，所以这里的签名是把变参传给 `format_sql`）

- [ ] **`db_result_to_json_array(MYSQL_RES *res)`**（第318-367行，定义在 api.h 但实现在 api.c）：
  - 遍历每一行 → 对于每一行遍历所有字段 → 根据 `MYSQL_FIELD.type` 判断是否为数值类型 → 数值不加引号，字符串加引号并转义 → 拼成 `[{"col1":"val1","col2":123},...]` 格式
  - `is_numeric_field()` 判断：TINY/SHORT/LONG/LONGLONG/INT24/FLOAT/DOUBLE/DECIMAL/NEWDECIMAL → 数值
  - **结果用 `malloc` 分配，调用方负责 `free()`**
  - NULL 输入 → 返回 `"[]"`

- [ ] **为什么连接前要执行 `SET NAMES utf8mb4`？**
  数据库存储 utf8mb4，但 MySQL 默认客户端连接字符集是 latin1。不设置会导致中文写入时乱码、读取时显示为 `?`。代码在 `create_connection` 中做了两层设置：`mysql_options(MYSQL_SET_CHARSET_NAME)` + `mysql_set_character_set()`。

#### 1.2 认证层 `server/src/common/auth.c` + `include/auth.h`

- [ ] **会话表结构**（第45行）：
  ```c
  static Session g_sessions[MAX_SESSIONS];  // MAX_SESSIONS = 1024
  ```
  每个 Session 包含：`in_use`（是否被占用）、`token`（33字节）、`user_id`、`last_access`（最后访问时间戳）。
  **这是一个固定大小的数组，不是链表或哈希表**，查找方式是 O(n) 线性扫描。对于单机部署、少量用户（几百人）足够。

- [ ] **`auth_init()`**（第46-50行）：初始化互斥锁 → `memset` 清零全部会话 → 用 `time(NULL)` 初始化随机种子

- [ ] **`auth_login(username, password, out_user, out_token)`**（第108-175行）：
  1. 校验用户名/密码非空 → 用 `db_escape()` 转义用户名
  2. `SELECT` 用户（使用 `USER_SELECT_COLS` + `USER_SELECT_FROM` 宏，带 LEFT JOIN 学院/专业/班级）
  3. `fill_user_from_row()` 填充 User 结构体
  4. `utils_password_verify(password, u.password)` 校验密码（SHA256 比对）
  5. 检查 `u.status != 1` → 账号禁用返回 ERR_STATUS
  6. `UPDATE users SET last_login=NOW()` 更新最后登录时间
  7. `INSERT INTO logs` 记录登录日志
  8. `utils_gen_random_string(token, 32)` 生成 32 位随机 token
  9. **会话槽位分配**：加锁 → 找空闲槽 → 槽位满则回收最久未访问的（LRU 驱逐）→ 写入 token/user_id/last_access → 解锁

- [ ] **`fill_user_from_row(User *u, MYSQL_ROW row)`**（第59-81行）：
  - **列顺序必须与 `USER_SELECT_COLS` 宏严格一致**
  - 逐个字段用 `atoi(row[n])` 或 `utils_strlcpy` 填充
  - 对 NULL 字段用 `utils_safe()` 转为空串

- [ ] **`USER_SELECT_COLS` 宏**（第84-88行）：
  定义了标准用户查询的 SELECT 列（19 列），包含 users 表的全部字段 + LEFT JOIN 的 college_name/major_name/class_name。`fill_user_from_row` 依赖这个列顺序。

- [ ] **`auth_logout(token)`**（第177-190行）：加锁 → 遍历会话表 → 匹配 token → 标记 `in_use = 0` → 解锁

- [ ] **`auth_resolve_session(token, out_user)`**（第192-212行）：
  1. 加锁 → 遍历找匹配 token → 取 user_id → 更新 last_access → 解锁
  2. `auth_load_user(uid, out_user)` **每次都从数据库重新加载用户信息**（保证角色/状态实时生效）
  3. 检查 `out_user->status != 1` → 会话期间被禁用的立即失效

- [ ] **设计关键：每次请求都重新查库**
  为什么要这样？因为内存中的 User 可能已过时——比如管理员禁用了这个用户、或者修改了角色。每次都查库保证 **管理操作即时生效**，不需要额外的"踢人下线"机制。

#### 1.3 工具函数 `server/src/common/utils.c` + `include/utils.h`

- [ ] **SHA256 密码哈希（第26-145行）**：
  - 完整实现了 FIPS 180-4 标准的 SHA256 算法，无需 OpenSSL 依赖
  - 核心结构：`SHA256_CTX { state[8], bitlen, data[64], datalen }`
  - `sha256_init()`：初始化 8 个 32 位初始哈希值（标准规定的魔数，如 `0x6a09e667`）
  - `sha256_update()`：将输入数据按 64 字节分块送入 `sha256_transform()`
  - `sha256_transform()`：64 轮迭代，每轮涉及 CH/MAJ/EP0/EP1/SIG0/SIG1 等位运算（Sha256 核心算法）
  - `sha256_final()`：填充（pad）→ 追加长度 → 最后一遍 transform → 输出 32 字节原始哈希
  - `utils_sha256(input, out_hex)`：init → update → final → 32 字节转为 64 字符十六进制 + `\0`
  - `utils_password_verify(plain, hash)`：对明文计算 SHA256 → 用 `strcmp` 与已存哈希比对

- [ ] **输入校验函数**：
  - `utils_validate_student_no(s)`：必须 10 位纯数字
  - `utils_validate_id_card(s)`：18 位，前 17 位数字，末位可为 X/x。**选填字段**（空值放行返回 1）
  - `utils_validate_phone(s)`：11 位纯数字，以 `1` 开头
  - `utils_validate_username(s)`：4-50 位，仅字母数字下划线
  - `utils_is_empty(s)`：NULL 或全空白字符 → 返回 1

- [ ] **安全字符串操作**：
  - `utils_strlcpy(dst, src, size)`：类似 BSD 的 `strlcpy`，保证 `dst` 以 `\0` 结尾
  - `utils_trim(s)`：原地去除首尾空白，用 `memmove` 将中间内容前移
  - `utils_str_equal(a, b)`：安全比较，处理 NULL 指针情况（NULL == NULL 返回 1，NULL vs 非NULL 返回 0）

- [ ] **日期时间**：
  - `utils_now_datetime(out)`：`time()` → `localtime()` → `strftime("%Y-%m-%d %H:%M:%S")`
  - `utils_now_date(out)`：同上取日期部分
  - `utils_add_days(date_in, days, date_out)`：`sscanf` 解析 → `mktime` 加减 → `strftime` 输出
  - `utils_datetime_compare(a, b)`：直接用 `strcmp` 比较（因为 `YYYY-MM-DD HH:MM:SS` 格式天然满足字典序 = 时间序）

- [ ] **随机生成**：
  - `utils_gen_signin_code(out)`：6 位随机数字，用于活动签到码（`rand() % 10`）
  - `utils_gen_random_string(out, len)`：从 62 个字符集（大小写字母+数字）中随机选取，用于生成 Token
  - `ensure_seed()`：用 `time(NULL) ^ clock()` 异或作为种子，只初始化一次（`g_rand_seeded` 标志）

## 第二阶段：API 框架层（必须精通）

### 2.1 入口与请求分发 `server/src/main.c`

- [ ] **启动顺序**（第294-351行 `main()`）：
  1. 打印横幅 → `load_config("db_config.ini")` 读取数据库配置（有默认值兜底，文件不存在也能启动）
  2. `db_init(&cfg)` 初始化连接池（失败则 `fprintf stderr` + `return 1`）
  3. `auth_init()` 初始化会话池（清零 + 初始化锁）
  4. `signal(SIGINT/SIGTERM, signal_handler)` 注册信号处理（Ctrl+C 优雅退出）
  5. `mg_mgr_init(&mgr)` 初始化 Mongoose 事件管理器
  6. `mg_http_listen(&mgr, "http://0.0.0.0:8000", ev_handler, NULL)` 监听所有网卡的 8000 端口
  7. **事件循环**：`while (s_signo == 0) mg_mgr_poll(&mgr, 200)` 每 200ms 轮询一次
  8. 收到信号 → `mg_mgr_free` → `auth_cleanup` → `db_close` → 退出

- [ ] **配置加载 `load_config()`**（第54-96行）：
  - 先设置全部默认值（`127.0.0.1:3306`，`root`，`scms_db`，pool=5）
  - 逐行读取 INI 文件：跳过空行/注释行（`#`/`;` 开头），`[section]` 识别段名，`key=value` 解析键值
  - 用 `trim_line()` 去除首尾空白和 `\r\n`
  - 文件打不开只打印 WARN，不会导致启动失败

- [ ] **事件处理函数 `ev_handler`**（第259-290行）：
  - 只处理 `MG_EV_HTTP_MSG` 事件（HTTP 请求）
  - **OPTIONS 请求** → 返回 204 + CORS 头（`Access-Control-Allow-Origin: *`、`Methods: GET,POST,PUT,DELETE,OPTIONS`、`Headers: Content-Type`）
  - **`/api/*` 前缀** → `dispatch_api(c, hm)` 进入业务路由分发
  - **`/receipts/*` 前缀** → `mg_http_serve_dir(c, hm, {.root_dir="storage"})` 提供发票图片
  - **其他** → `mg_http_serve_dir(c, hm, {.root_dir="public"})` 提供前端 Vue SPA 静态文件

  **关键理解**：这个项目不需要 Nginx！Mongoose 作为一个嵌入式 HTTP 库，直接处理了 API 路由、静态文件服务、CORS、Cookie 解析等所有 HTTP 层功能。`scms.exe` 就是完整的 Web 服务器。

- [ ] **Cookie 解析 `get_token_from_cookie()`**（第101-110行）：
  - 用 `mg_http_get_header(hm, "Cookie")` 获取 Cookie 头
  - 用 `mg_http_get_header_var(cookie, mg_str("scms_token"))` 提取名为 `scms_token` 的 Cookie 值
  - 这是 Mongoose 内置的工具函数，不需要手写 Cookie 解析

- [ ] **URI 匹配 `uri_match()`**（第117-133行）：
  - `*` 表示**匹配单一路径段**（不跨越 `/`）
  - 例：`pattern="/api/school/clubs/*"` 匹配 `/api/school/clubs/12`，不匹配 `/api/school/clubs/12/members`
  - 实现原理：逐字符比对，遇到 `*` 吃掉 URI 中直到下一个 `/` 的内容
  - `method_match()`：比较 HTTP 方法字符串（GET/POST/PUT/DELETE）

- [ ] **三个内置路由**（不走路由表，直接写在 `dispatch_api` 中）：
  - **`POST /api/login` → `handle_login()`**：优先从 JSON body 取 username/password，退化到表单。调用 `auth_login()` → 成功则 Set-Cookie 下发 token + 返回用户信息 JSON。登录成功也在 `auth_login()` 内部记录日志
  - **`POST /api/logout` → `handle_logout()`**：取 Cookie 中的 token → `auth_logout()` 销毁会话 → `Set-Cookie Max-Age=0` 清除浏览器 Cookie
  - **`GET /api/me` → `handle_me()`**：返回当前 `ctx->user` 的信息（已在 dispatch_api 中解析好）。前端用这个接口做登录状态判断和角色校验（在导航守卫中调用）

- [ ] **请求分发 `dispatch_api()`**（第212-255行）：
  1. 构建 `ApiContext`：填充 conn、hm、method
  2. 从 Cookie 解析 token → `auth_resolve_session()` 还原用户 → 写入 `ctx.user`
  3. 先匹配三个内置路由
  4. 遍历路由表（`routes_table(&n)`），逐个 `method_match` + `uri_match`
  5. 首个匹配的检查 `need_login`：若需要登录而 `ctx.user == NULL` → 返回"请先登录"
  6. 调用 `routes[i].handler(&ctx)`
  7. 没有任何匹配 → 返回"接口不存在"

- [ ] **CORS 处理**：OPTIONS 预检返回三个关键头部，`Access-Control-Allow-Origin: *` 允许跨域（开发时 Vite dev server 在 :5173 端口，API 在 :8000 端口）

- [ ] **信号处理**：`volatile int s_signo` → Ctrl+C 触发 `SIGINT` → `signal_handler` 设置标志 → 事件循环退出 → 依次清理 mgr/auth/db

### 2.2 API 辅助层 `server/src/common/api.c` + `include/api.h`

这是业务模块开发最常用的文件，封装了所有 HTTP 交互细节。

- [ ] **`ApiContext` 结构体**（api.h 第29-34行）：
  ```c
  typedef struct {
      struct mg_connection *conn;   // Mongoose 连接句柄
      struct mg_http_message *hm;   // HTTP 请求消息（含 method/uri/headers/body）
      User *user;                   // 当前登录用户（未登录为 NULL）
      char method[8];               // HTTP 方法字符串（GET/POST/PUT/DELETE）
  } ApiContext;
  ```

- [ ] **参数读取函数（重点理解）**：

  **`api_get_param(ctx, key, out, out_size)`**（第42-59行）：
  - **先查 query string**：`mg_http_get_var(&ctx->hm->query, key, out, out_size)` — 适用于 GET 请求
  - **再查 POST body**：`mg_http_get_var(&ctx->hm->body, key, out, out_size)` — 适用于 `application/x-www-form-urlencoded` POST
  - **值写入 out 缓冲区**（不是返回值！返回值是 0/1 表示找到/未找到）
  - 这是之前踩过坑的函数——错误用法 `char *status = api_get_param(ctx, "status", "")` 会导致编译错误

  **`api_get_int(ctx, key, def_val)`**（第61-66行）：
  - 内部调用 `api_get_param` → `atoi()` 转换，未找到返回 `def_val`

  **`api_get_json_str(ctx, key, out, out_size)`**（第75-89行）：
  - 用于前端以 `Content-Type: application/json` 提交的请求
  - 内部用 Mongoose 的 `mg_json_get_str(hm->body, "$.key")` 从 JSON 中提取字段
  - 返回值是 `malloc` 的 → `utils_strlcpy` 到 out → `free`

  **`api_get_path_int(ctx, idx)`**（第111-139行）：
  - 从 URL 路径中提取整数参数
  - 例如路由 `/api/clubs/12/members`，`idx=1` 取到 `12`（跳过 `/api/` 前缀后按 `/` 分段，从 0 开始计数）
  - 实现：跳过 `/api/` → 逐段跳过 `seg_start` 到 `/` → 匹配 idx → `atoi` 返回

  **`api_get_form_str(ctx, key, out, out_size)`**（第147-164行）：
  - 用于 multipart/form-data 表单（含文件上传时，不能用 `api_get_param`）
  - 用 `mg_http_next_multipart()` 循环遍历 body 中的 part → 按 name 匹配 → 复制 body 到 out

- [ ] **JSON 响应函数（全部 HTTP 状态码为 200）**：

  - `api_send_json(ctx, json)`：最底层，直接 `mg_http_reply(200, Content-Type+CORS头, json)`
  - `api_ok(ctx)` → `{"code":0,"msg":"ok"}`
  - `api_ok_msg(ctx, msg)` → `{"code":0,"msg":"<msg>"}`，msg 用 `MG_ESC()` 宏做 JSON 转义
  - `api_ok_data(ctx, data_json)` → `{"code":0,"msg":"ok","data":<data_json>}`，data_json 直接拼接不转义
  - `api_error(ctx, code, msg)` → `{"code":<code>,"msg":"<msg>"}`，HTTP 状态仍为 200

  **设计要点**：所有响应的 HTTP 状态码都是 200！前端通过 JSON 中的 `code` 字段判断成功/失败（`code === 0` 成功）。这样做的好处是前端统一用 `if (res.code === 0)` 判断，不需要分别处理 HTTP 200/400/403/500 等情况。

- [ ] **结果集自动转 JSON 响应**：

  - `api_send_result(ctx, res)`：`db_result_to_json_array(res)` → 自动 `mysql_free_result` → `api_ok_data`
  - `api_send_result_data(ctx, res)`：与上面完全相同（别名，代码中冗余但保持兼容）
  - `api_send_result_paged(ctx, res, page, page_size, total)`：在 JSON 中外层额外加上 `page`/`page_size`/`total`/`total_pages` 分页字段

  使用模式：`api_send_result(ctx, db_query("SELECT ..."))` 一行搞定查询+返回！

- [ ] **权限检查函数**：
  - `api_require_login(ctx)`：检查 `ctx->user != NULL`，未登录返回"请先登录"
  - `api_require_role(ctx, role)`：检查 `ctx->user->role` 是否等于指定角色

### 2.3 路由系统 `server/src/common/routes.c` + `include/routes.h`

- [ ] **`Route` 结构体**（routes.h 第32-37行）：
  ```c
  typedef struct {
      const char *method;    // "GET"/"POST"/"PUT"/"DELETE"
      const char *pattern;   // URI 模式，* 匹配单段
      ApiHandler  handler;   // 处理函数指针
      int         need_login; // 1=需登录
  } Route;
  ```

- [ ] **路由表组织**（routes.c）：
  - 静态数组 `ROUTES[]`，编译时确定，约 117 条
  - 按角色分组：公共(4) → 通知(4) → 学校(约33) → 学院(约20) → 学生(约21) → 社团管理(约35)
  - `routes_table(&n)` 返回数组首地址和条数

- [ ] **路由排序规则（重要）**：
  - **具体路径必须在通配路径之前**！例如 `GET /api/school/clubs/pending` 必须在 `GET /api/school/clubs/*` 之前
  - 原因：`dispatch_api` 的 for 循环取**第一个匹配**的路由。如果 `*` 在前，`pending` 会被 `*` 吞掉，永远匹配不到
  - 同样：`POST /api/clubs/apply` 在 `POST /api/clubs/*/join` 之前

- [ ] **路径参数提取**：
  - `api_get_path_int(ctx, n)` 中 n 从 0 开始，是 `/api/` 之后的段序号
  - 例：路由 `/api/club/*/activities/*`，匹配 `/api/club/5/activities/12`
    - `n=0` → `"club"`（跳过，字符串）
    - `n=1` → `5`（社团 ID）
    - `n=2` → `"activities"`（跳过）
    - `n=3` → `12`（活动 ID）

- [ ] **`need_login` 机制**：
  - `need_login=1` 的路由，`dispatch_api` 会在调用 handler 前检查 `ctx->user`
  - `need_login=0` 的路由（4 条公开接口）直接调用
  - 角色权限检查在 handler 内部用 `api_require_*` 系列函数做细粒度控制
  - 这种设计让路由层只做粗粒度的"需登录/不需登录"，权限检查留给业务层

### 2.4 桩函数机制 `server/src/common/handlers_stub.c`

- [ ] **弱符号与强符号覆盖**：
  - `handlers_stub.c` 中所有 handler 都定义为**普通强符号函数**
  - 当业务模块的 `.c` 文件通过 `#if 1` 启用后，其同名 handler 函数编译成目标文件 `.o`
  - 链接时，同名的强符号会产生"重复符号"错误
  - 所以启用模块时需要**注释掉对应的 `STUB_HANDLER(...)` 行**（注释中有明确说明）

  实际工作流程：
  1. 刚开始所有模块 `#if 0` → 所有 handler 走 stub（返回"功能尚未实现"）
  2. 开发者写好后将 `#if 0` 改成 `#if 1` → 注释对应 `STUB_HANDLER` 行 → `make`
  3. 如果忘记注释 STUB_HANDLER → 链接报错 `duplicate symbol`

- [ ] **`STUB_HANDLER(name)` 宏**（第27-28行）：
  ```c
  #define STUB_HANDLER(name) void name(ApiContext *ctx) { stub_reply(ctx); }
  ```
  展开后生成一个返回 `{"code":-99,"msg":"该功能尚未实现（业务模块待编写）"}` 的函数

- [ ] **当前项目的模块状态**：所有业务模块的 `#if 0` 都已改为 `#if 1`（已全部开发完成），所以所有 handler 都从真实的 `.c` 文件中获得实现，stub 只是保底编译用的占位

### 5.2 认证流程
- [ ] **登录全链路**：
  1. 前端 `api.post('/api/login', {username, password})`
  2. 后端 `handle_login` → `auth_login` → SHA256 校验 → 生成 token → Set-Cookie 下发
  3. 浏览器自动保存 Cookie（HttpOnly + SameSite=Lax）
  4. 后续所有 `api.xxx()` 请求自动携带 Cookie（`credentials: 'include'`）

- [ ] **导航守卫**（`router/index.js`）：
  1. 用户访问受保护页面 → `router.beforeEach` 触发
  2. 调用 `GET /api/me` 检查登录状态
  3. 后端 `handle_me` 从 Cookie 还原用户 → 返回 {user_id, username, role, ...}
  4. 前端根据 `AREA_ROLES` 判断当前角色是否有权访问该路由
  5. 无权 → 重定向到角色首页；通过 → 放行

## 第六阶段：数据库要点（理解即可）

### 6.1 核心表关系
- `users` → `colleges`（学生归属学院）
- `clubs` → `colleges`（社团归属学院）+ `users` via `president_id`
- `members` ↔ `clubs` + `users`（多对多，一个学生可在多个社团）
- `activities` ↔ `clubs` → `registrations` ↔ `users`（活动报名）
- `reimbursement` ↔ `clubs`（报销归属社团）
- `elections` ↔ `clubs` → `election_candidates` ← `election_votes`

### 6.2 5 个触发器
1. `trg_member_after_insert`：成员批准 → `clubs.member_count + 1`
2. `trg_member_after_update`：pending→approved → +1；设置 left_at → -1
3. `trg_registration_after_insert`：活动报名 → `activities.current_count + 1`
4. `trg_registration_after_update`：取消报名 → `activities.current_count - 1`
5. `trg_reimbursement_after_update`：报销批准 → `reimbursement_limits.current_period_used + amount`

**为什么用触发器**：保证数据一致性。无论哪个后端 handler 修改这些表，计数器都会自动同步，不会因为忘记手动更新而出错。

### 6.3 ENUM 字段速查
| 表 | 字段 | 可选值 |
|----|------|--------|
| users | role | school_admin / college_admin / club_admin / club_member / general_student |
| clubs | status | pending / approved / rejected / archived / dissolved |
| clubs | level | school / college |
| members | role | president / vice_president / secretary / treasurer / member |
| members | join_status | pending / approved / rejected |
| activities | status | draft / published / ongoing / completed / cancelled / archived |
| registrations | status | registered / cancelled / attended |
| finance | type | income / expense |
