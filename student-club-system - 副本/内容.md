
## 基础设施层（所有业务模块的必经之路）

以下4个模块是任何业务功能都绕不开的，类似于操作系统的内核：

| 模块 | 定位 | 为什么绕不开 |
|------|------|-------------|
| api.c | HTTP请求/响应工厂 | 参数解析 + JSON构建 + 登录/角色校验，所有118条路由都依赖它 |
| db.c | 数据库唯一入口 | 连接池管理 + SQL执行 + 结果集转JSON，无它则无数据读写 |
| routes.c | 路由分发器 | 注册118条路由 + URL通配符匹配 + 分发给对应handler |
| auth.c | 会话管理 | 登录/登出/token校验 + 1024槽位哈希表存储会话 |

## URL通配符匹配引擎 — 为什么前端路由做不到

### 问题场景

后端有118条API路由，大量使用带动态ID的路径：
```
/api/school/clubs/1/approve   (审批社团1)
/api/school/clubs/12/approve  (审批社团12)
/api/club/5/announcements     (社团5的公告)
/api/college/students/203/restrict (限制学生203)
```

### 为什么不用前端路由

前端 Vue Router 是**客户端路由**，只能操作浏览器地址栏和页面跳转。它完全无法：
1. 解析 `POST /api/school/clubs/12/approve` 中的 `12` 提取为参数
2. 把 `GET /api/club/5/announcements` 分发给不同的后端处理函数
3. 根据 HTTP 方法区分同一个路径的不同操作（GET读/POST写/PUT改/DELETE删）

这是**服务端路由**的工作，必须有C代码来干。

### 手写匹配引擎（main.c + routes.c）

```c
// main.c: uri_match — 仅12行，O(n)复杂度
static int uri_match(const char *pattern, struct mg_str uri) {
    const char *p = pattern;
    const char *u = uri.buf;
    const char *uend = uri.buf + uri.len;
    while (*p) {
        if (*p == '*') {
            p++;  // 跳过 '*'
            while (u < uend && *u != '/') u++;  // 吃掉当前段
        } else {
            if (u >= uend || *p != *u) return 0;
            p++; u++;
        }
    }
    return (u == uend) ? 1 : 0;
}
```

### 关键设计

| 设计点 | 说明 |
|--------|------|
| *通配符 | 匹配单一路径段（不跨/），对应动态ID |
| 路由排序 | 具体路径排在通配路径前面，防止 /school/clubs/pending 被 /school/clubs/* 误吞 |
| method_match | 同时匹配HTTP方法，GET和POST到同一URL可分发给不同handler |
| api_get_path_int | 按 /api/ 后的段序号提取动态参数值，如 ctx,2 取 club_id |

### 实际分发流程

```
POST /api/school/clubs/12/approve
  → uri_match("/api/school/clubs/*/approve", ...) = 1
  → method_match("POST", ...) = 1
  → need_login = 1 → 校验登录态
  → 调用 sch_club_approve(ctx)
  → ctx内 api_get_path_int(ctx, 2) 返回 12
```

### 前端路由做不到的原因总结

前端 Vue Router 只管"用户看到哪个页面"，后端路由引擎管"请求数据落到哪个函数"。118条API + 动态ID + HTTP方法区分 + 登录态校验，这些是前端路由完全无法插手的事。这个12行的 `uri_match` 是整个系统的"流量调度中心"。
