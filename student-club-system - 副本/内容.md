
## routes.c 路由匹配流程

### 整体架构

```
HTTP请求到达
  → dispatch_api() 解析Cookie还原用户会话
  → 先匹配内建接口: /api/login, /api/logout, /api/me
  → 再遍历业务路由表(118条): routes_table() → ROUTES[]
  → method_match() + uri_match() 双重匹配
  → 找到首个匹配项 → 调用 handler(ctx)
```

### 两步匹配

1. **method_match** — HTTP方法匹配 (GET/POST/PUT/DELETE), 同一URL不同方法分发给不同handler
2. **uri_match** — 12行C实现，逐字符匹配，`*`通配单路径段(不跨`/`)，对应动态ID

### 路由排序规则

具体路径必须在通配路径之前注册，否则被误吞：
```
/api/school/clubs/pending    ← 具体路径先注册
/api/school/clubs/*/approve  ← 通配路径后注册
/api/school/clubs/*          ← 兜底匹配最后
```

### 动态参数提取 (api_get_path_int)

handler内通过段序号提取动态值：
```
POST /api/school/clubs/12/approve
  → uri_match("/api/school/clubs/*/approve", ...)
  → api_get_path_int(ctx, 2) → 12  (从0开始: 0=school, 1=clubs, 2=12)
```

### 完整分发流程

```
POST /api/school/clubs/20/approve
  1. dispatch_api 解析请求,Cookie还原token → 获取school_admin会话
  2. 匹配内建接口 → 不匹配(非/login/logout/me)
  3. 遍历ROUTES表:
     - "/api/school/clubs/pending" → uri_match=? 不匹配
     - "/api/school/clubs/history" → uri_match=? 不匹配
     - "/api/school/clubs/*/approve" → uri_match=1, method_match(POST)=1
     - need_login=1, ctx.user!=NULL → 通过
  4. 调用 sch_club_approve(&ctx)
  5. handler内 api_get_path_int(ctx,2) → 20
  6. 执行审批逻辑,返回JSON
```

## 典型Bug排查流程

### Bug1: 学院仪表盘学生人数显示0（数据库列名错误）

**现象：** cs_admin登录后仪表盘"本院学生数"始终显示0，但学生管理页面有27人

**排查过程：**
1. curl仪表盘API → `student_count=0`
2. curl学生管理API → `total=27` → 确认不是数据缺失
3. 对比两处SQL:
   - 仪表盘: `WHERE college_id=%d AND is_active=1` ← 可疑
   - 学生管理: `WHERE college_id=%d AND status=1 AND role IN (...)`
4. `SHOW CREATE TABLE users` → 只有`status`列，没有`is_active`列
5. 根因: `is_active`是不存在的列名，MySQL静默返回0

**修复:** `is_active=1` → `status=1 AND role IN ('general_student','club_member','club_admin')`

**教训:** 数据库列名不能凭记忆，需先DESCRIBE确认；两处统计口径必须一致

### Bug2: 学生管理页面分页按钮显示禁止图标（前端props缺失）

**现象：** 信科院学生管理页面27人分3页，翻到第2页后翻页按钮变灰色禁止

**排查过程：**
1. 查后端API → `total=27, total_pages=3` 正常，排除后端问题
2. 查前端DataTable组件 → `currentPage=ref(props.page)`, `currentSize=ref(props.size)`
3. 查StudentMgmt.vue → 只传了`:data/:columns/:total`，**没传`:page/:size`**
4. 用户翻页 → emit `page-change` → StudentMgmt更新`page.value=2`
5. 但DataTable内部`currentPage`始终是初始值1 → `el-pagination`检测到`currentPage`与实际激活页不一致 → 灰掉翻页按钮
6. 根因: DataTable的`currentPage`通过ref一旦初始化就不再同步外部变化，必须用props传入

**修复:** StudentMgmt.vue添加`:page="page" :size="10"`传给DataTable

**教训:** Vue组件props和内部ref需要双向绑定才能同步；分页组件的current-page必须与外部状态保持联动
